BOM Pipeline Core /bom-pipeline-core

Shared pipeline: Deal → Planset → BOM → Zoho Sales Order. Canonical steps used by planset-to-so (single job) and bom-so-analysis (batch). Handles deal lookup, planset auto-selection, BOM extraction, customer matching, and SO creation.

deal-to-SO pipeline referenced by planset-to-so referenced by bom-so-analysis
.claude/skills/bom-pipeline-core/
Claude Vision HubSpot Google Drive Zoho Inventory

Input

  • PROJ number (e.g., "PROJ-9009")
  • Customer name (e.g., "Turner")
  • HubSpot deal ID

Pipeline Steps

1. Find Deal in HubSpot (auto-select if single match) ↓ 2. Auto-Select Planset (Stamped Plans > recent > "stamped" filename) ↓ 3. Extract BOM (planset-bom skill, all sheets) ↓ 4. Save BOM Snapshot (POST /api/bom/history, auto-increment version) ↓ 5. Auto-Match Zoho Customer (hubspot_contact_id → name search) ↓ 6. Create Sales Order (POST /api/bom/create-so)

HubSpot Properties Fetched

dealname, hs_object_id, design_documents, all_document_parent_folder_id, project_number, associated_contact_id, module_count, roof_type, system_size_kwdc

Escalation Model

ErrorDefault Action
Deal not foundEscalate — ask caller for correct identifier
No PDFs in design folderEscalate — provide Drive folder link
BOM validation failsEscalate — caller decides continue/fix
No Zoho customer matchEscalate — caller provides customer ID
SO creation failsLog error, escalate
Unmatched items on SOLog them, continue (skipped)

"Escalate" = return control to caller. planset-to-so asks the user; bom-so-analysis logs and skips to next job.

SO Response Fields

  • salesorder_number — Zoho SO number (e.g., SO-00456)
  • unmatchedItems[] — BOM items that couldn't match Zoho inventory
  • corrections[] — post-processor changes (sku_swap, item_removed, qty_adjust, item_added)
  • jobContext — detected: jobType, roofType, moduleCount, hasPowerwall, etc.

Planset BOM /planset-bom

Extract a complete Bill of Materials from a Photon Brothers stamped solar/storage planset PDF. Outputs CSV, Markdown, and JSON matched to the EquipmentSku inventory schema.

read a planset generate a BOM extract equipment from a planset what's on this planset read this design plan
v0.1.0.claude/skills/planset-bom/
Claude Vision / PDF

Workflow

Planset PDF ↓ 1. Read PDF (PV-0 cover, PV-2 BOM table, PV-4 SLD) ↓ 2. Extract project header (system size, customer, AHJ) ↓ 3. Parse BOM table (equipment, qty, description) ↓ 4. Extract conductor schedule + rapid shutdown from SLD ↓ 5. Run validation (module count, battery kWh, OCPD) ↓ 6. Output: Markdown + CSV + JSON

BOM to Sales Order /bom-to-so

Convert a saved BOM snapshot into a draft Zoho Sales Order with automatic post-processing, then optionally compare against the ops-created SO.

create SO from BOM generate sales order compare auto vs ops SO what does the post-processor do
v0.1.0.claude/skills/bom-to-so/
HubSpot Zoho Inventory

Prerequisites

  • Required BOM snapshot saved via planset-bom skill
  • Required Zoho customer ID resolved (auto-match or manual)

Pipeline

1. Resolve Zoho Customer (hubspot_contact_id → name fallback) ↓ 2. POST /api/bom/create-so { dealId, version, customerId } ↓ Loads BOM → matches Zoho items → runs SO post-processor ↓ 3. Verify: SO number, unmatched items, corrections ↓ 4. Fetch ops SO: GET /api/bom/zoho-so?so_number=SO-XXXX ↓ 5. Compare: normalize, match by SKU/name, classify mismatches

Planset to Sales Order /planset-to-so

End-to-end: Deal → Planset → BOM → Zoho Sales Order for one project. Delegates steps 1–5 to bom-pipeline-core, adds a user confirmation checkpoint before SO creation.

process PROJ-XXXX create SO for this deal run the full pipeline SO for [customer]
.claude/skills/planset-to-so/
Claude Vision HubSpot Google Drive Zoho Inventory

Required Sub-Skill

/bom-pipeline-core — handles steps 1–5 (deal → planset → BOM → customer match). This skill adds the user confirmation checkpoint before SO creation.

Workflow

1. Run pipeline steps 1–5 from bom-pipeline-core ↓ 2. Show BOM summary (module count, battery type, racking, validation) ↓ 3. If pipeline escalates → ask the user ↓ 4. Show SO preview ← ONE required checkpoint ↓ 5. On approval → run pipeline step 6 (create SO) ↓ 6. Report result (SO number, unmatched items, corrections)

SO Preview Format

  • Deal name, Zoho customer ID, job type
  • Line items table with qty and source (BOM vs post-processor)
  • Post-processor corrections summary (SKU swaps, additions, removals)
  • User confirms or rejects before creation

What This Skill Does NOT Do

  • Does not run batch analysis (use /bom-so-analysis)
  • Does not bypass user confirmation before SO creation
  • Does not handle PO creation (separate workflow)

BOM SO Analysis /bom-so-analysis

Batch analysis of auto-generated vs ops-created Sales Orders. Orchestrates pipeline runs, fetches SOs from Zoho, classifies jobs, compares line items, and proposes post-processor improvements.

run the SO analysis compare auto vs ops SOs find SO patterns improve post-processor batch compare SOs
v0.1.0.claude/skills/bom-so-analysis/
Claude Vision HubSpot Google Drive Zoho Inventory

Prerequisites

  • Dev server running (for API access)
  • ENABLE_SO_POST_PROCESS=true in env
  • BOM tool feedback fetched first

Batch Workflow

1. Select jobs (PROJ numbers, warehouse filter, or all) ↓ 2. Per job: Run bom-pipeline-core (ALWAYS fresh, never reuse old snapshots) ↓ 3. Fetch auto SO + ops SO from Zoho, compare line items ↓ 4. Classify all jobs (type, warehouse, equipment) ↓ 5. Pattern analysis (frequency, qty formulas, breaker patterns) ↓ 6. Propose post-processor rule improvements ↓ 7. Write findings to ~/Downloads/SOs/

Pattern Analysis Dimensions

  • Item frequency by warehouse × job type
  • Qty formulas vs module count (HUG, MCI-2, screws, clamps, etc.)
  • Breaker brand patterns (HOM260, Q260, BR260) by region
  • Missing items in >80% of ops SOs = candidate post-processor rules
  • Enphase vs Tesla equipment differences
  • Service upgrade trigger identification

Required Skills

  • /bom-to-so — Reference for post-processor rules and comparison methodology
  • /product-lookup — Manufacturer reference data for validating quantities, SKU variants, and installation rules during pattern analysis

  • session-N-analysis.md — Findings, frequency tables, proposed rules
  • PROJ-XXXX/ops-so-data.md — Per-job ops SO data
  • PROJ-XXXX/comparison-notes.md — Per-job auto vs ops comparison

Look up solar equipment specifications, installation quantities, sizing rules, and compatibility from curated manufacturer reference files. Web search fallback for products not yet documented.

how many snow dogs per array? what clamp for this module frame? XR10 vs XR100? PW3 expansion kit contents
v0.1.0.claude/skills/product-lookup/
Web Search

When to Use

  • BOM extraction — determining correct quantities (snow dogs per array, MCI-2 per module)
  • SO creation — selecting the right SKU variant for a roof type, panel brand, or inverter platform
  • User questions — clamp ranges, load capacities, torque specs
  • SO analysis — validating ops-created quantities against manufacturer guidelines

Workflow

1. Parse query — identify manufacturer, product, info needed ↓ 2. Read reference files (check Manufacturer Reference Index) ↓ 3. Answer from reference — cite source file ↓ 4. Web fallback — search manufacturer docs if references insufficient ↓ 5. Flag gaps — note missing products for future addition

Find Design Plans /find-design-plans

Locate planset PDFs and design documents for any PB project. Full lookup chain: HubSpot deal → Google Drive folder → Stamped Plans subfolder.

find a planset PDF look up design documents planset for PROJ-XXXX
v0.1.0.claude/skills/find-design-plans/
HubSpot Google Drive

Lookup Chain

1. Search HubSpot for deal (by name or PROJ number) ↓ 2. Extract folder ID from design_documents URL ↓ (fallback: all_document_parent_folder_id → "2. Design" subfolder) ↓ 3. GET /api/bom/drive-files?folderId={ID} ↓ (recursive breadth-first PDF scan) ↓ 4. Return file list: { id, name, mimeType, modifiedTime, size }

Drive Folder Structure

[Project Root] 0. Sales/ 1. Site Survey/ 2. Design/ ← design_documents points here Stamped Plans/ ← planset PDFs DA/ Archive/ 3. Permitting/ 4. Interconnections/ 5. Installation/ 6. Inspections/ 7. PTO & Closeout/ 8. Incentives/

HubSpot Properties

PropertyUsage
design_documentsDirect URL to "2. Design" folder (preferred)
all_document_parent_folder_idRoot project folder ID (fallback)

Auth Note

/api/bom/drive-files requires a logged-in user session (Google OAuth). In chat, user must share a direct Drive file link or use the BOM tool UI at /dashboards/bom?deal=DEAL_ID.

Debug Zuper field service integration issues — API problems, job syncing, scheduling mismatches, and status comparison discrepancies.

Zuper API problems job syncing issues scheduling mismatches status discrepancies

Zuper MCP HubSpot

API Architecture

Route GroupPurpose
zuper/jobs/Job CRUD and scheduling
zuper/jobs/schedule/Schedule operations

zuper/status-comparison/HubSpot ↔ Zuper status sync
zuper/availability/Technician availability

Key Gotchas

  • assigned_to is CREATE-only — Zuper API ignores it on updates
  • Custom fields differ — GET returns array, POST expects object
  • Status field — Use current_job_status, not status
  • Category matters — Each category has separate status workflows
  • Team/User UIDs — JSON env vars: ZUPER_TEAM_UIDS, ZUPER_USER_UIDS

Debug Steps

  • Identify scope: which API route, job category, or status?
  • Check API route in src/app/api/zuper/
  • Check types in src/lib/zuper.ts
  • Query live data via Zuper MCP tools
  • Compare with HubSpot via status-comparison route

Zuper MCP Tools

  • list_jobs / get_job_details — Query live Zuper data
  • list_job_categories — Get category UIDs and status workflows
  • get_module_filters — Discover filterable fields
  • get_job_team_stats / get_job_stats_category — Aggregated stats

Release Notes /release-notes

Generate release notes from git commits for preparing a release, changelog, or deployment summary.

release notes prepare changelog deployment summary
No AI model.claude/skills/release-notes/
Git only

Output Format

## Release Notes — [date] ### New Features - [from feat: commits] ### Bug Fixes - [from fix: commits] ### Improvements - [refactors, performance, UX] ### Infrastructure - [build, deploy, CI]

Rules

  • Group commits by type (feat, fix, refactor, chore)
  • Write user-friendly descriptions (not raw commit messages)
  • Combine related commits into single bullet points
  • Highlight breaking changes at the top
  • Skip merge commits and trivial changes
  • Include dashboard names if mentioned in commits
  • Defaults to commits since last tag, or last 20 if no tags
