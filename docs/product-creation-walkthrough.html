Product Creation Walkthrough

End-to-end trace of what happens when a user submits a new product through the wizard at /dashboards/submit-product, showing the record that lands in InternalProduct, HubSpot, Zoho Inventory, and Zuper, plus the cross-link IDs that connect them all. Example throughout: a Silfab 410W module being submitted by a sales rep.

1

User opens the wizard

User

Sales rep, ops, or anyone authed clicks "Submit New Product" from the Operations suite or hits /dashboards/submit-product directly. The wizard loads.

Auth: any authenticated user can reach the wizard (Tested: SALES, OPS, ADMIN). Per Phase B decisions, this is intentional — the wizard supports a sales-rep-driven request flow. Non-admin submissions get an audit-flagged ActivityLog row.

URL params can pre-fill the wizard if you arrive from a deal context: ?dealId=PROJ-1234&category=MODULE&brand=Silfab&model=SIL-410-HC+. This skips Start Mode and lands you directly on the Basics step.

2

Pick a start mode

User

The wizard offers three starting paths:

  • Start from scratch — empty form. Best for adding a brand-new product with no reference.
  • Clone existing — search the catalog by SKU/model. Selecting a result pre-fills basics + specs + vendor. Useful for variants (different wattage of same module line, etc.).
  • Import from datasheet — upload a manufacturer PDF spec sheet. Claude extracts brand, model, category, and spec fields. Output goes through handleDatasheetExtracted → wizard pre-filled.

Whichever path the user picks, they land on the Basics step.

8

Auto-approval engine starts

Server

The route handler immediately calls executeCatalogPushApproval(push.id, { source: "wizard", userEmail: authResult.email }) from src/lib/catalog-push-approve.ts. The function:

  1. Captures startedAt = Date.now()
  2. Locks the PendingCatalogPush row (must be PENDING)
  3. Initializes per-system outcomes map with { status: "skipped", message: "Pending processing." }
  4. Walks each selected system in sequence (Steps 9–12 below)
  5. Writes cross-link IDs (Step 13)
  6. Uploads photo to Zoho (Step 14)
  7. Finalizes status (Step 15)
  8. Writes ActivityLog row (Step 16)

HUBSPOT block — create or update Product

HubSpot API

createOrUpdateHubSpotProduct(...) is called. It:

  1. Searches HubSpot Products by hs_sku for an existing match
  2. If none, searches by (name, manufacturer, product_category)
  3. If still none, POSTs a new Product to /crm/v3/objects/products
  4. If manufacturer isn't in HubSpot's enum AND HUBSPOT_MANUFACTURER_ENFORCEMENT=true: PATCHes the enum to add the brand, retries the create, notifies TechOps via email
  5. Returns { hubspotProductId, created }

ZOHO block — create or update Inventory item

Zoho API

createOrUpdateZohoItem(...) is called. It:

  1. Lists active Zoho items (cached) and matches by SKU → part_number → name
  2. If existing: PATCHes vendor + description + part_number + group/category
  3. If new: POSTs to /items with full payload (uses category_id from the new getZohoCategory registry — for MODULE = 5385454000001229316)
  4. Writes cf_internal_product_id on the item
  5. Returns { zohoItemId, created }

ZUPER block — create or update Product

Zuper API

createOrUpdateZuperPart(...) is called. It:

  1. Searches existing Zuper Products by SKU/model/name
  2. If new: POSTs to /product with brand, model, sku, description, unit, prices, category UID (resolved via resolveZuperCategoryUid), specification string ("410W Mono PERC"), and customMetaData spec entries via buildZuperSpecMetaData

14

Photo upload to Zoho

Zoho API

If metadata._photoUrl was set (the user uploaded an image), uploadZohoItemImage(zohoId, bytes, fileName, contentType) POSTs the bytes to /items/{item_id}/image.

The photo URL stored on metadata is an internal viewer URL like /api/catalog/photo?path=catalog-photos%2Fsilfab-410.png — the engine extracts the blob path and fetches the bytes from Vercel Blob server-side, then forwards to Zoho.

Best-effort, Zoho-only: failure to upload doesn't fail the overall approval — appends a warning to the Zoho outcome message. HubSpot and Zuper photos are not pushed (out of scope; tracked in followups).

Alternate entry points

The wizard isn't the only way products enter the catalog. Three other paths exist, all eventually run through the same executeCatalogPushApproval engine (or its building blocks):

BOM pipeline (deal-driven)

When a planset PDF is parsed and an extracted item doesn't match an existing InternalProduct via bom-catalog-match.ts, a PendingCatalogPush is created automatically with source: "bom_pipeline", reviewReason: "no_match", and a 90-day TTL. Sits in PENDING until an admin approves via /dashboards/catalog?tab=pending. Approval calls the same executeCatalogPushApproval(id, { source: "approval_retry" }).

Sync Modal (per-product, ad-hoc)

From the catalog table (/dashboards/catalog), opening Sync Modal on a row lets ADMIN/EXEC users link an existing external record OR create a new one for any single system. Goes through executePlan in catalog-sync-plan.ts with source: "modal". After the create, writeCrossLinkIds fires so the new external record isn't an orphan — same helper as the wizard path uses.

Direct API upsert (admin/scripts)

For data-cleanup work, POST /api/inventory/products can upsert an InternalProduct row directly without touching external systems. ADMIN/OWNER/PROJECT_MANAGER only. Used by ops scripts (e.g., the Zoho orphan reconciliation backfill). External cross-links can be passed in via hubspotProductId / zohoItemId / zuperItemId; if any are passed, an ActivityLog row records the manual link.
